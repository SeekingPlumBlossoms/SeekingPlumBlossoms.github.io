分布式全局唯一ID生成策略

分布式系统的定义

  • 分布式系统就是对外是统一的入口，对内是有很多计算机服务组装的系统。

为什么需要一个全局唯一主键

  • 主要是用于某个记录的存储或者输出的时候，能找到唯一的记录，不能让数据错乱。数据库的唯一ID在单库单表的情况下可以使用，但是在使用分布式系统的时候，数据库的自增主键就不可以使用了.举个栗子：先有订单记录表1024张,如果使用数据库中的自增主键，那么就会有1024订单号相同的订单，这个肯定是不行的，所以需要分布式唯一的主键去确定订单号。

分布式唯一主键的需求

  1. 最基本也是最重要的：全局唯一，不可重复
  2. 有一定的连续性:数据库存储使用的是B-TREE数据结构存储，我们要尽量保证ID是有序的增长
  3. 主键上需要能看到数据的记录位置

分布式唯一主键生成的方式

1.UUID生成唯一主键

  • 优点
    1. 使用方便，性能高，无网络或者IO消耗.
  • 缺点
    1. 无法作为DB主键使用，但是可以对于分布式链路追踪ID可以适用
    2. 长度太长：36位

2.使用Redis生成唯一的主键

  • 使用redis的原子incr操作，保证实现主键递增；
  • 优点
    1. 不依赖数据库，性能高
    2. 主键自然增长，能均匀分布到数据库中
  • 缺点
    1. 需要引入redis并且需要维护自增的主键key,自增到什么长度数据合适(需要考虑)
    2. 对于redis性能有要求

3.数据库自增ID机制

  • 依据插入数据库返回的数据主键进行计算出主键,即是需要逻辑的索引库，库的作用就是获取序列。
  • 优点
    1. 简单有序。充分利用数据库的自增
  • 缺点
    1. 生成ID需要读写数据库
    2. 对数据库的性能保障有要求

4.SnowFlake雪花算法

  • 优点
    1. 简单高效,不依赖外部组件
    2. 按照时间有序递增
    3. 12位序列：毫秒内的计数，12位的计数顺序号支持每个节点每毫秒(同一机器，同一时间截)产生4096个ID序号
  • 缺点
    1. 依赖机器的时钟，如果时钟回拨，会造成ID重复
    2. 分布式环境下，机器时钟不是同步，会出现ID不是递增的情况
